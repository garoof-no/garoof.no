## Abstractions as things with holes in them

There's a Quora question about abstraction that has ^https://www.quora.com/What-is-the-difference-between-simplification-and-abstraction/answer/Panicz-Godek an answer by Panicz Godek that I really like^.

After reading that I've thought that "an abstraction is a thing with holes in it" is often a useful way to think about it. Should read the Quora answer, but like, in case Quora is dead or too enshittified by now and this post is somehow still alive: The first example there uses a pair of "find the differences" pictures. And then a third picture is an "abstraction" that is the same as the two others where those two pictures are equal but has "holes" where the two pictures are different. So the abstraction is something we can use to create either of the first to pictures (as well as other pictures) by filling in the holes. It's a very visual example that now tends to spring to my mind when like "the nature of abstractions" or something is brought up.

## Functions

In something like ^../lambs/lambs.txt lambda calculus^, we can think of bound variables as holes that can be filled in. If we have, say, a function with a function in it:

```
λa.λb.a b a
```

Then with a couple of function applications:

```
(λa.λb.a b a) foo bar
```

We can like fill in those holes/variable references:

```
foo bar foo
```

In many programming languages, we can think of at least some of the function calls in the same way, and that can be a useful mental model. See also ^https://mitp-content-server.mit.edu/books/content/sectbyfn/books_pres_0/6515/sicp.zip/full-text/book/book-Z-H-10.html#%_sec_1.1.5 SICP: The Substitution Model for Procedure Application^.

Let's pretend we only have the number zero and successor and predecessor functions in Lua:

``` run
function s(n) return n + 1 end
function p(n) return n - 1 end
function positive(n) return n > 0 end
```

We could make some numbers:

``` run
four = s(s(s(s(0))))
six = s(s(four))
five = p(six)
```

``` repl
print(four, five, six)
```

And let's say we wanna add two of them together, like maybe four and six. We'll start at four and then like count up from there while counting down from six:

``` repl
local result, additional = four, six
while positive(additional) do
  result = s(result)
  additional = p(additional)
  print(result, additional)
end
print(result);
```

If we want to add together _five_ and six instead, we can go like this:

``` repl
local result, additional = five, six
while positive(additional) do
  result = s(result)
  additional = p(additional)
  print(result, additional)
end
print(result);
```

Those two pieces of code are similar and we can make an abstraction that kind of works for both by turning the differences into holes. We'll make a function that takes the addends as parameters.

``` run
function foo(a, b)
  local result, additional = a, b
  while positive(additional) do
    result = s(result)
    additional = p(additional)
    print(result, additional)
  end
  print(result);
end
```

``` repl
foo(four, six)
foo(five, six)
```

Let's clean it up and make it return the result:

``` run
function add(a, b)
  while positive(b) do
    a = s(a)
    b = p(b)
  end
  return a
end
```

``` repl
print(add(four, six))
print(add(five, six))
```

We can get our original "addition results" by putting in the appropriate arguments. So that's cool. We can also use other arguments:

``` repl
print(add(four, five))
print(add(add(four, five), six))
```

So that's an abstraction. It's not concretely _the result of adding two specific numbers together._ It has holes in it, and we can put many different numbers in those holes. In this particular case, since numbers go in the whole, and we also get a number back when we fill in the holes, we can put the result of one addition into the hole of another. It's pretty cool tbh.

In a typical programming language we have have "the same hole" multiple times:

``` run
function triple(n)
  return add(n, add(n, n))
end
```

Passing in one number will make it appear in all the three places:

``` repl
print(triple(four))
```

So yeah.

## It is possible to think about some things

I think it's reasonably clear how we can think of function abstractions as "things with holes in them" like this, particularly if they are very functional functions. It can be kind of visual, like with the "find the differences" pictures: We can look at the `triple` function and go, oh right we have this "incomplete expression" that goes `add(__, add(__, __)`, and we can "complete" it by choosing a number that will go in all the `__`-holes.

There are some ways functions can be and not be that can make more easy or less easy to think of them as these kinds of abstractions.

* The `triple` function is a very functional function. We can look at its code and use the substitution model: Replace the references to `n` with the number passed in and get our result.
* The `foo` function is less satisfying. Doing the substitution thing works. But it prints its result as a side effect, so we cannot easily get the result back and use it for other stuff.
* I like that, kind of on a techincality, the substitution model doesn't fully work for the `add` function. Since we're assigning to the arguments in the while loop, it doesn't make sense to like substitute e.g. `four` for `a` and in `a = s(a)` and get `four = s(four)`. In practice it behaves pretty functionally though. The way Lua works, parameters are basically local variables and these assignments cannot be observed from outside the function.

## Generalities, flags, ...

I think it's fair to say that addition is ^./generalities.txt more general^ than one number that happens to be the sum of two numbers, and so on.

Thinking of functions as "things with holes in them" might suggest one way boolean flags can feel a little "wrong." There's something direct about "find the differences" picture with holes in it. If we think of it as a function it makes sense to think that we should apply it to a number of tiny pictures that will go in the holes. We could imagine a similar abstraction that took a number of boolean values instead, with true meaning that we should use that part from the first of the two different pictures and false meaning we should use that part from the second. I do think the visual picture with the visible holes suggests "tiny pictures as arguments" and that that feels more direct or something.

Functions are sometimes a little like this:

``` run
function square_(x, y, size, yellow)
  return
    '<svg width="' .. (x + size + 8) .. '" height="' .. (y + size + 8) .. '">'
    .. '<rect x="' .. x .. '" y="' .. y .. '"'
    .. ' width="' .. size .. '" height="' .. size .. '"'
    .. ' stroke="' .. (yellow and "yellow" or "white") .. '" />'
    .. '</svg>'
end
```

You know, it makes squares:

``` repl
web.html(square_(30, 40, 50, true))
web.html(square_(10, 20, 80, false))
```

And like okay it has a hole that can be filled with true or false, but we're like, yeah and by true we mean yellow and by false we mean white. Like that hole feels more properly filled with the actual thing once we have kind of evaluated the `(yellow and "yellow" or "white")` bit in our head for the boolean value we're passing in. When we consider a function call with its arguments, something like `true` feels more indirect than something like `50`.

We could do this instead:

``` run
function square(x, y, size, colour)
  return
    '<svg width="' .. (x + size + 8) .. '" height="' .. (y + size + 8) .. '">'
    .. '<rect x="' .. x .. '" y="' .. y .. '"'
    .. ' width="' .. size .. '" height="' .. size .. '"'
    .. ' stroke="' .. colour .. '" />'
    .. '</svg>'
end
```

I think the meaning of those arguments become a little more obvious when seeing a function call:

``` repl
web.html(square(30, 40, 50, "yellow"))
web.html(square(10, 20, 80, "white"))
```

I think talk about abstractions, particularly "too much abstraction" and "too general" stuff, can sometimes give the impression that we're turning a knob. Like how abstract _and_ general _and_ indirect do we want it to be? But like there are different ways to go about things. I think going from booleans to colours makes it more general but also less indirect. More general in that you can make squares with outlines in many more different colours, and more direct in that uh, you directly refer to the color, not to some value that will be used by the function to choose a colour.

A traditional `map` function is more general than a similar function that takes a boolean or another enumerated type that is used to decide which operation should be performed on all the elements. Passing in the operation you want makes it more general and it also makes it more direct: You don't have to open up the implementation of `map` to see which operation it chooses if you pass in `true`, or `5` or `"GCD"` or something. The operation is referred to right there, and you can either see it there as a lambda function or you can find it with very regular "go to definition" stuff that is not concerned with how it will be used in the `map` function.

And so on. I think "general and direct" is often good for doing reusability without making a _complete_ mess of things.

## Inversion of control

If the `add` function "depends on" two numbers then we can say that it has its dependencies "injected." We can "inject" numbers like four and five by passing them in as arguments, and the function body of `add` isn't concerned with how those numbers got there. It doesn't need to construct those numbers and it doesn't need to go find them somewhere.

We could imagine an addition function that does not have its dependencies injected but instead looks for them itself, in a specific place. We'll have the place be a "container:"

``` run
local Container = {
  getBean = function(t, k) return t[k] end,
  setBean = function(t, k, v) t[k] = v end
}
applicationContext = setmetatable({}, { __index = Container })
```

And then we can make an addition function that does not use dependency injection:

``` run
function addBean()
  local a = applicationContext:getBean("a")
  local b = applicationContext:getBean("b")
  while positive(b) do
    a = s(a)
    b = p(b)
  end
  return a
end
```

Now we can call the function with different inputs by putting different numbers in the container.

``` repl
applicationContext:setBean("a", four)
applicationContext:setBean("b", five)
print(addBean())

applicationContext:setBean("a", six)
print(addBean())
```

It's not exactly a way to live but like it's something you could do. Building a "container" would be a very strange thing to do if you wanted to encourage dependency injection.

## Methods

Method calls tend to be kind of like function calls. In some languages they are exactly that. E.g. in Lua, if we take a string method like `gsub`, it's just a function, but when we call it "like a method" the "receiver of the method call," like the "this" or "self" object, is passed in as a first argument. These two `gsub` calls are pretty equivalent:

``` repl
local str = "ababa"
print(str:gsub("a", "-"))
local f = str.gsub
print(f(str, "a", "-"))
```

In other languages there can be a little more to it than that. Either way, in many cases methods are similar to functions in how they work or don't work as "things with holes in them abstractions."

## Objects

In blue book DDD terms I think it's usually more useful to think about value object methods as abstractions like this and less useful to think about methods on entity objects that way. When we're dealing with immutable value/data objects, it makes sense to think of passing arguments to a constructor as filling the holes of that abstract thing.

^https://mastodon.social/@romeu/116319147937559106 Christian Queinnec/Romeu Moura says^:

> It was Christian Queinnec’s quote that "passing arguments to a constructor is partial application"
> 
> The unstated assumption (and humor) of the quote is that for that to be true you need immutable objects with no side effects

The constructed object is _kind of like_ a set of functions that already have some of their arguments passed in. If we're building value objects in a "composition over inheritance" kind of way, some of those already passed in arguments will be objects with their own methods (and with their own already passed in arguments).

## Classes

Abstract class definitions are kind of class definitions with holes left in them for some of the method bodies. Incomplete classes that can be completed by filling the holes.

Classes with virtual methods in general are like, every implemented and virtual method is:

* Like a hole, in that we can put in a method implementation by overriding it.
* And like not a hole, in that nothing's missing. We don't have to put in a method implementation but can use it as it is.

Here's ^https://lists.squeakfoundation.org/pipermail/squeak-dev/1998-April/009261.html a comment from Alan Kay on inheritance^. I've seen others. A couple of quotes:

> confusions of Taxonomy and Parentage, of Specialization and Refinement, of Parts and Wholes, of Semantics and Pragmatics.

...

> All this makes
> `    Foo subClass: #Bar`
> a difficult line of code to understand -- even if you really understand Foo, it is not at all clear what Bar will be like. It could be very much like Foo or not at all. Inheritance is a weak notion in most OOP languages.

I think it's easy to kind of accidentally come up with classes that are like, when we view them as "things with holes in them" abstractions, it's not really spelled out like "and these are the holes that should be filled." Sometimes it's like if we think of one set of methods as holes then that works as one abstraction, but if we choose a different set of methods it works as another abstraction. It might not be obvious which sets of methods it makes sense to override at the same time, like if overriding one typically leads to inconsistency unless we also override this other one.

## Larger things

A library will typically consist of like functions, classes, objects and methods and stuff. Some of them will make more sense as "things with holes in them" abstractions, while others might be less like that. Frameworks are weirder, obvs. I guess some frameworks are kind like, incomplete software that can be completed by putting like "the application business domain stuff" into the holes. In practice I think they usually make it very unclear which holes exist, where they are and what they're shaped like, and how they're filled.

## Blah blah

I guess like, there's something about in what way it does and/or does not make sense, or is or is not easy, to think of something as a "thing with holes in it" abstraction. Like that says something, about something. Like viewing a pure and functional function that way is fairly straightforward:

* It is clear where the holes are and how they are filled.
* By applying the function to arguments we fill all the holes _in one go_ and get a completed thing back, and that completed thing is a final and immutable thing.

That might have something to do with _some kind of_ simplicity. With an "entity object," or a fairly large and classical "OO class," or a framework, it might be less straightforward to consider it an abstraction _in that sense._ I'm not sure an "X is simpler than Y" take makes sense here, but I dunno, maybe like those other things are less simple in that particular way. Something along those lines...

Anyway, I like to decide: If something seems to be useful as _that kind of abstraction,_ then I wanna kind of emphasize that. Make it a function, make if pure and functional, make the data immutable. But if not then I maybe won't. If what I want is an entitiy object, with a "life cycle" and with methods that change its state, then I won't make those methods return stuff. E.g I wouldn't make a "fluent interface" for that kind of stuff: That would muddy the waters and I'd only get a very tiny amount of syntactical conveniece in return. Stuff like that.
