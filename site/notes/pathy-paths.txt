## """problem statement"""

So if we're using something like Disjktra's algorithm on something like a tilemap we tend to get one of the minimally expensive paths, according to some way of calculating costs. The mental model could be something like a turn based strategy game: There could be roads and different terrain types having different movement costs and more expensive does not need to mean more moves made or more positions visited.

In these examples things will be fairly simple:

* Terrain is intraversable (`#`) or intraversable (typically space characters)
* Movement cost is 2 for orthogonal movement and 3 for diagonal

Anyway it can be the case that a lot of different paths from one position to another have the same cost. A thing that's been on my mind is something about picking a path that looks kind of "natural." Particularly with stuff like: If we're moving to a position and in order to get there we have to move east 10 times and diagonally to the northeast 3 times, we might get paths like:

```
            o
           o
          o
oooooooooo
``` 10 moves west followed by 3 moves to the northwest

While if we're making the moves by hand we might make a path that looks more like one line (despite both paths being equally good wrt cost). Maybe something like:

```
          ooo
       ooo
   oooo
ooo
``` 3 west, 1 northwest, 4 west, 1 northwest, 3 west, 1 northwest, 3 west

So something to do with that has been on my mind. How would I go about choosing paths that look less "artificial," when the artificial movement cost rules make the "artifical" ones technically as good as the "natural" ones?

## a found path

The data structure we get from the `flood` function also has information about how we got to it. When we look up a position and get a result if it's reachable, the result we get will contain a `from` table, with directions as keys and positions as values. A way to find a path to a position is to start with that position and then step back towards where we started the flood filling by always following the first `from` value we find:

``` run
function stepback(p, fill)
    local seen = fill[p]
    if not seen then return nil end
    for _, d in ipairs(dirs) do
      if seen.from[d] then return seen.from[d] end
    end
    return nil
end

function steppy(str, stepback)
  local map = mappy(str)
  local start, stop = next(map.S), next(map.E)
  local q = flood(map, start)
  local path = { [stop] = true }
  local p = stepback(stop, q.seen, start)
  while p do
    path[p] = true
    p = stepback(p, q.seen, start)
  end
  local function drawpath(c, p)
    return path[p] and c == " " and "o" or c
  end
  web.html(maphtml(map, drawpath))
end
```

Now we can find one path:

``` repl
steppy(complicated, stepback)
```

## aside: leaving the flood fill alone

When I'm trying out different stuff here, I'm always using the same flood filling part and I'm only changing the part where I construct a path from its result. In _some_ cases it might kind of make more sense to make modifications to the part that searches from the start position instead of to the part that constructs the path from the end position. E.g. the stepback function above checks for allowed diagonal steps before orthogonal steps (which makes the diagonal movement come last, since we're stepping from the end position when constructing the path). We can make an alternative version of stepback that does it the other way around:

``` run
function altstepback(p, fill)
    local seen = fill[p]
    if not seen then return nil end
    for i = #dirs, 1, -1 do
      local d = dirs[i]
      if seen.from[d] then return seen.from[d] end
    end
    return nil
end
```

Which should make the diagonal movement come first:

``` repl
steppy(simple, altstepback)
```

Often changes like that are made to the initial search part instead of to the later construct a path part and in a number of cases that can make more sense. But for some of the stuff I wanted to try I needed all the possible paths or all the positions hit by all the possible paths, so here it seemed easier to do the flood fill stuff the same way for all the things and only make changes to the construct the path part. ## a probably bad path comparison idea

Looking at the two paths from earlier again. One goes almost all the way to the east and then turns and goes notheast:

```
            o
           o
          o
oooooooooo
``` 10 moves west followed by 3 moves to the northwest

The other goes a few steps to the east then one step northeast then a few steps east and so on:

```
          ooo
       ooo
   oooo
ooo
``` 3 west, 1 northwest, 4 west, 1 northwest, 3 west, 1 northwest, 3 west

If we prefer the latter then: Maybe the best pats are the ones with the shortest sequences of movement in the same direction. Like the first one has consecutive steps to the east, while the last one has at most 4. An idea could be to find the best path by checking all the alternatives when backstepping and choosing the paths with the shortest sequences of same-direction steps.

There's some stuff that could be done to the code below but I dropped the idea before doing more so uh it is what it is. Also won't get into too much details. When a path back to the start is found, we make a list of all the "segments" and sort that list. We remember the best path found so far and compare new ones to that one. Anyway the code:

``` run
local function better(a, b)
  for i = 1, math.min(#a, #b) do
    if a[i].count < b[i].count then return true end
    if a[i].count > b[i].count then return false end
  end
  return (#a < #b) and a or b
end

local function best(path, currentbest)
  local prev = path[#path]
  local list, stuf = { prev }, {}
  for i = #path - 1, 1, -1 do
    local p = path[i]
    list[#list + 1] = p
    local dir = prev - p
    if stuf[#stuf] and stuf[#stuf].dir == dir then
      stuf[#stuf].count = stuf[#stuf].count + 1
    else
      stuf[#stuf + 1] = { dir = dir, count = 1 }
    end
    prev = p
  end
  table.sort(stuf, function(a, b) return a.count > b.count end)
  local res = { path = list, stuf = stuf }
  if not currentbest then return res end
  return better(stuf, currentbest.stuf) and res or currentbest
end

local function findbest(fill, start, stop)
  local pathcount = 0
  local path = { stop }
  local currentbest
  local function halp()
  local p = path[#path]
    if p == start then
      pathcount = pathcount + 1
      currentbest = best(path, currentbest)
      return
    end
    local seen = fill[p]
    if not seen then return end
    local i = #path + 1
    for _, pp in pairs(seen.from) do
      path[i] = pp
      halp()
    end
    path[i] = nil
    return
  end
  halp()
  return currentbest.path, pathcount
end 

function bestpath(str)
  local map = mappy(str)
  local start, stop = next(map.S), next(map.E)
  local fill = flood(map, start).seen
  local path, count = findbest(fill, start, stop)
  print("found " .. count .. " paths")
  for _, p in ipairs(path) do
    map[p] = "o"
  end
  web.html(maphtml(map))
end
```

## rectangles

If we look at the positions in all of the possible minimally expensive paths between two positions, the "problem areas" appear to be some fairly rectangular areas of positions. After flood filling and choosing a stop position, we can kind of flood fill back to the start positions by visting all the positions we can reach through the `from` values:

``` run
function fillback(fill, p)
  local set, visited = { [p] = true }, { [p] = true }
  while next(set) do
    local p = next(set)
    set[p] = nil
    local seen = fill[p]
    for _, d in ipairs(dirs) do
      local from = seen.from[d]
      if from and not visited[from] then
        set[from] = true
        visited[from] = true
      end
    end
  end
  return visited
end
```

Now we can look at the rectangle:

``` repl
local map = mappy(simple)
local start, stop = next(map.S), next(map.E)
local q = flood(map, start)
local back = fillback(q.seen, stop)
local function heat(c, p)
  if not back[p] then return c end
  c = c == " " and "." or c
  return heatcoloured(c, p, q)
end
web.html(maphtml(map, heat))
```

So the kind of rectangular area has a bunch of positions in it that can be reached through at least one of the minimally expensive paths. In this case most of them can be reached directly from the position to the west and the position to the southwest. IMO paths look more "natural" if the look more like a line from the southwest corner to the northeast corner of the rectangle. So that's a thing.

## maybe a less bad idea?

Here's a function to see how far off we are:

``` run
function off(from, to, p)
  local fx, fy, tx, ty, x, y, dx, dy
  local d = to - from
  if math.abs(d.x) >= math.abs(d.y) then
    fx, fy, tx, ty, x, y, dx, dy = from.x, from.y, to.x, to.y, p.x, p.y, d.x, d.y
  else
    fx, fy, tx, ty, x, y, dx, dy = from.y, from.x, to.y, to.x, p.y, p.x, d.y, d.x
  end
  if dx == 0 then return math.abs(y - fy) end
  local target
  if fx == x then target = fy
  else target = fy + (((x - fx) * dy + (dx // 2)) // dx)
  end
  return math.abs(y - target)
end
```

That is, if we imagine a line between two points, `from` and `to`, then this should give us the number of steps we need to go east or west or north or south in order to get to be on top of the line. It rotates some numbers around a bit to account for lines that are more vertical than horizontal vs. lines that are more horizontal than vertical. Uuuh:

``` repl
local from, to = vec(0, 0), vec(8, 3)
for y = 0, 3 do
  for x = 0, 8 do
    io.write(off(from, to, vec(x, y)))
  end
  io.write("\n")
end
```

Anyway the idea is that if you're trying to move kind of along a line and you have to decide between a couple of positions, then you wanna choose the one with the lowest off-number.

So a plan can go something like:

We use the `fillback` function from earlier to get like a pretend map that might have rectangles in it. And then when we're constructing the path and are stepping from one position to the next (or previous, I guess), we look into some things:

* What rectangle are we in? We'll make a pretty low-effort attempt at determining.
* Are we choosing between exactly two "ajacent" directions. (E.g. nort and northwest, or east and southeast.)
* What would be good positions for a line to go from/to?
* Which move gets us closer to the line?

We do the rectangle check by "shooting" rays through the "backfilled" data structure that potentially has rectangles in it. We shoot one ray in each of the four main directions and see how far it goes. We could do something more clever instead but this seems to be mostly okay. We use the directions available for moving to determinge which corners of the rectangle the imagined line should go from/to. Some stuff:

``` run
local function shoot(p, dir, back)
  local prev = p
  while back[p] do
    prev = p
    p = p + dir
  end
  return prev
end

function rectback()
  local back
  return function(p, fill)
    if not back then back = fillback(fill, p) end
    local seen = fill[p]
    if not seen then return nil end
    local from = seen.from
    local function regular()
      local _, res = next(from)
      return res
    end
    if seen.dirs ~= 2 then return regular() end
    local n, e, s, w = shoot(p, N, back).y, shoot(p, E, back).x, shoot(p, S, back).y, shoot(p, W, back).x
    local start, stop
    local d1, d2
    if from[W] and from[NW] then
      d1, d2 = W, NW
      start, stop = vec(e, s), vec(w, n)
    elseif from[N] and from[NW] then
      d1, d2 = N, NW
      start, stop = vec(e, s), vec(w, n)
    elseif from[N] and from[NE] then
      d1, d2 = N, NE
      start, stop = vec(w, s), vec(e, n)
    elseif from[E] and from[NE] then
      d1, d2 = E, NE
      start, stop = vec(w, s), vec(e, n)
    elseif from[E] and from[SE] then
      d1, d2 = E, SE
      start, stop = vec(w, n), vec(e, s)
    elseif from[S] and from[SE] then
      d1, d2 = S, SE
      start, stop = vec(w, n), vec(e, s)
    elseif from[S] and from[SW] then
      d1, d2 = S, SW
      start, stop = vec(e, n), vec(w, s)
    elseif from[E] and from[SW] then
      d1, d2 = E, SW
      start, stop = vec(e, n), vec(w, s)
    else return regular()
    end
    local p1, p2 = from[d1], from[d2]
    return (off(start, stop, p1) <= off(start, stop, p2)) and p1 or p2
  end
end

## appendix: the dijkstrarian flood fill stuff
:id flood-fill

``` prelude
function memo(key, new)
  local all = setmetatable({}, { __mode = "v" })
  return function(...)
    local k = key(...)
    local found = all[k]
    if found then return found end
    local v = new(...)
    all[k] = v
    return v
  end
end

Vec = {}
vec = memo(
  function(x, y) return x .. "," .. y end,
  function(x, y) return setmetatable({ x = x, y = y }, Vec) end)

function Vec.__add(a, b) return vec(a.x + b.x, a.y + b.y) end
function Vec.__sub(a, b) return vec(a.x - b.x, a.y - b.y) end
N, E, S, W = vec(0, -1), vec(1, 0), vec(0, 1), vec(-1, 0)
NW, NE, SE, SW = N + W, N + E, S + E, S + W
function Vec.__tostring(v) return v.x .. "," .. v.y end
dirs = { NW, NE, SE, SW, N, E, S, W }

Lazy = {}
function Lazy.__index(t, k)
  local list = {}
  t[k] = list
  return list
end

Queue = {}
Queue.__index = Queue
function Queue.new()
    local q = {
        highest = 0,
        lowest = 0,
        seen = {},
        lists = setmetatable({}, Lazy)
    }
    setmetatable(q, Queue)
    return q
end

function Queue.put(q, pos, cost, dir, from)
  local seen = q.seen[pos]
  if seen then
    if seen.cost == cost then
      seen.from[dir] = from
      seen.dirs = seen.dirs + 1
      return
    elseif seen.cost < cost then
      return
    end
    local prev = q.lists[seen.cost]
    local current = prev.next
    while current do
      if current.pos == pos then
        prev.next = current.next
        break
      end
      local next = current.next
      prev = current
      current = next
    end
  end
  q.seen[pos] = { cost = cost, from = dir and { [dir] = from } or {}, dirs = dir and 1 or 0 }
  q.lists[cost].next = { pos = pos, next = q.lists[cost].next }
  q.highest = math.max(q.highest, cost)
end

function Queue.get(q)
  local i = q.lowest
  for i = q.lowest, q.highest do
    local list = q.lists[i]
    if list.next then
      local entry = list.next
       list.next = entry.next
       local pos = entry.pos
       q.lowest = i
       return pos, i
    end
  end
end

function mappy(str)
  local lines = str:gmatch("[^\n]+")
  local map = {}
  local y = 0
  local w = 0
  for line in lines do
    y = y + 1
    local x = 0
    for c in line:gmatch(".") do
      x = x + 1
      local v = vec(x, y)
      if c == "#" then
        map[v] = c
      end
      local set = map[c] or {}
      set[v] = true
      map[c] = set
    end
    w = math.max(w, x)
  end
  map.size = vec(w, y)
  return map
end

function maphtml(map, f)
  local html = { '<pre><code>' }
  for y = 1, map.size.y do
    for x = 1, map.size.x do
      local p = vec(x, y)
      local c =
        (map.S[p] and 'S')
        or (map.E[p] and 'E')
        or map[p] or ' '
      html[#html + 1] = f and f(c, p) or c
    end
    html[#html + 1] = '\n'
  end
  return table.concat(html)
end

function neighbours(p, map)
  local i = 1
  local cost = 3
  return function()
    while i <= 8 do
      local dir = dirs[i]
      local np = p + dir
      if i == 5 then cost = 2 end
      i = i + 1
      if not map[np] then
        return np, cost, dir
      end
    end
    return nil
  end
end

function step(map, q)
  local p, cost = q:get()
  if not p then return false end
  for np, c, dir in neighbours(p, map) do
    q:put(np, cost + c, dir, p)
  end
  return true
end

function flood(map, start)
  local q = Queue.new()
  q:put(start, 0)
  local current = true
  while current do current = step(map, q, stop) end
  return q
end

function span(str, style)
  return '<span style="' .. style .. '">' .. str .. '</span>'
end

function heatstyle(n, max)
  local h = (1 - (n / max)) * 240
  return 'background-color: hsl(' .. h .. ', 100%, 50%)'
end

function heatcoloured(c, p, q)
  local seen = q.seen[p]
  return seen and span(c, heatstyle(seen.cost, q.highest)) or c
end
